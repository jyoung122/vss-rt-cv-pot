#### `_worker_loop()` (lines 266–288) — **primary loop rewrite**

Current model: single `await q.get()` + `await _process_job(job, pool)` —
serial by construction.

New model — concurrent fan-out bounded by semaphore:

- Acquire semaphore before dequeuing (so depth of `q.get()` calls ≤ N).
- Launch `_process_job` as a `asyncio.Task` (not awaited inline) so the loop
  can immediately dequeue the next job.
- Pass semaphore into `_process_job` so the task releases it in its `finally`
  block (after teardown).
- The worker loop becomes a task-spawner; individual `_process_job` tasks own
  their own completion lifecycle.

**Precise flow:**

```
while True:
    await sem.acquire()          # blocks when N tasks are running
    job = await q.get()          # dequeue next
    asyncio.create_task(_process_job(job, pool, sem))
    # do NOT await — loop spins back immediately
```

`_process_job`'s `finally` must call `sem.release()` after teardown (so the
semaphore slot opens only after NVStreamer and ffmpeg are cleaned up, not when
processing begins).

`q.task_done()` moves from `_worker_loop` to `_process_job`'s `finally` block
for the same reason.

**CancelledError handling:** the outer `asyncio.CancelledError` path in
`_worker_loop` (lines 279–281) must also cancel any still-running `_process_job`
tasks. Collect created tasks in a local list; on `CancelledError`, cancel and
gather them before returning. This ensures teardown (nvstreamer, ffmpeg) fires
even during shutdown.

#### `start_worker()` (lines 296–299)

Initialise the semaphore here, bound to the running event loop:

```python
global _semaphore
_semaphore = asyncio.Semaphore(MAX_CONCURRENT_STREAMS)
```

Pass `_semaphore` into `_worker_loop`.

#### `stop_worker()` (lines 303–313) — no structural change

Cancels `_worker_task`; the task's `CancelledError` handler (updated above)
propagates cancellation to all in-flight `_process_job` tasks.

## Concurrency model

```
                 asyncio.Queue (depth ≤ UPLOAD_QUEUE_MAX_DEPTH)
                       │
                       ▼
              _worker_loop() ←─ dequeues jobs, bounded by semaphore
                       │
          ┌────────────┼─────────────┐
          │            │             │
   _process_job    _process_job   (blocked: sem full)
   [video A]       [video B]
   │ rtsp_pub.start  │
   │ nvstreamer.reg  │
   │ plateau loop    │
   │ nvstreamer.unr  │
   │ rtsp_pub.stop   │
   └── sem.release ──┘
          │
   worker loop unblocks, dequeues next job
```

The semaphore is owned by the module (initialised in `start_worker`). `_worker_loop`
acquires one slot per job before dequeuing; `_process_job` releases the slot in
its `finally` block. This ensures the slot is held for the full lifetime of the
job — including teardown — so the next job never starts until the current one
has fully released all GPU-side resources.

At depth N+1: the `sem.acquire()` in `_worker_loop` blocks until a running job
completes. The queue holds the waiting job; `get_queue_position()` returns its
index correctly.

## Teardown order

### Happy path (plateau reached)

1. Plateau loop exits at `break` (line 239 area).
2. `pool.execute("UPDATE uploads SET dss_status='completed' ...")` — DB write.
3. Log `upload_queue.job.done`.
4. `finally`:
   a. `await nvstreamer.unregister_sensor(sensor_uuid)` — DELETE NVStreamer sensor.
   b. SDR/nvstreammux remove source (Branch A or B).
   c. `rtsp_publisher.stop(video_id)` — SIGTERM ffmpeg, wait up to 5s.
   d. `_active_job_ids.discard(video_id)`.
   e. `sem.release()`.
   f. `q.task_done()`.

### Hard timeout (plateau not reached)

Same as happy path, `dss_status='failed'`. Steps 4a–f identical.

### Exception in `_process_job`

1. Exception propagates to `except Exception` block (lines 253–261).
2. `pool.execute("UPDATE uploads SET dss_status='failed' ...")`.
3. Log `upload_queue.job.error` with `exc_info=True`.
4. `finally` runs steps 4a–f, same as above. Note: `sensor_uuid` may be `None`
   if the exception occurred before `nvstreamer.register_sensor` completed —
   the guard `if sensor_uuid is not None` prevents a spurious DELETE call.
   `rtsp_publisher.stop` is safe to call even if `start` was never called
   (it's a no-op when the video_id is not in `_publishers`).

### Shutdown (worker cancelled)

1. `_worker_loop` catches `CancelledError`.
2. Cancels all outstanding `_process_job` tasks (collected list).
3. `await asyncio.gather(*tasks, return_exceptions=True)` — each task's
   `CancelledError` triggers its own `finally` block, firing teardown.

## Per-video plateau watcher — DB pressure check

The current plateau loop issues one `SELECT count(*) FROM events WHERE video_id=$1`
per second per job (line 214 area). At N=2: 2 queries/sec. At N=4: 4 queries/sec.
Each query hits the existing primary key / index on `events.video_id`.

The plan's assessment (F3 note in the phase doc) is that this is fine up to
N=4. No code change is required. The F5 capacity probe will measure actual DB
latency under load and is the correct time to revisit if thrashing is observed.

**Verify:** `event_indexer.py:154` — `current_video_id = await r.get("current_video_id")` —
continues to read from Redis unchanged. F2 still writes the Redis key (one per
job, last writer wins under concurrency). This is acceptable for one release:
the indexer will tag all events with the video_id of whichever job wrote last,
which is incorrect under N>1. The correct fix is F1 (per-event sensorId
tagging). F2 must not attempt to fix this; the `MULTI_STREAM_ENABLED` flag
note in the phase plan covers the rollback rationale.

## Backwards compatibility

**CLAUDE.md guidance is "no shims, no feature flags"** unless there is an
explicit safety reason. The one exception identified in the phase plan is the
Redis `current_video_id` write, which the indexer reads. Rationale for keeping
it: if `nvstreamer.register_sensor` fails (NVStreamer down), the job degrades
to "no events indexed" rather than a crash. Removing the Redis write without
F1 in place would leave the indexer tagging every event with a stale video_id
from a previous run, which is worse. So:

- **Keep** the Redis `current_video_id` write (lines 180–183) unchanged in F2.
  No flag needed — it's not a backwards-compat shim for old callers, it's a
  required dependency until F1 lands.
- **Delete** `current_stream_url.txt` write (lines 184–187) — no callers remain
  once `_restart_container` is deleted. `ds-start.sh` reads it, but `ds-start.sh`
  is not called in the new flow (container is not restarted).
- **Delete** `_restart_container("vss-rt-cv")` call (line 190) and the
  `_docker_api` / `_restart_container` helpers (lines 126–153). No feature
  flag: the old single-stream path is completely replaced by the RTSP flow.

`_docker_api` and `_restart_container` have no tests (they were infrastructure
helpers), so deletion is clean.

## Open questions requiring operator decision

1. **SDR→DS handoff branch (A vs B):** parallel agent must report before the
   implementation PR is opened. This drives whether `sdr_client.py` is created
   or nvstreammux is called directly.

2. **What happens to in-flight jobs on `stop_worker` during a rolling deploy?**
   The current `stop_worker` cancels the task immediately. With long-running
   jobs (30–90s each), a deploy would kill them mid-stream. Options:
   - Wait for semaphore to drain before cancelling (graceful drain with timeout).
   - Accept the current behaviour (jobs fail, retry on next upload attempt).
   CLAUDE.md says no deferred wiring — a decision is needed now, even if the
   answer is "accept the current behaviour."

3. **`get_active_job_id()` API contract:** `uploads_list.py` calls this to
   set `queue_status` in the progress response. Under N-concurrent, "active"
   means "in `_active_job_ids`". The current per-job semantics (`active` vs
   `queued`) remain meaningful. But `get_active_job_id()` returning an arbitrary
   active job_id (not the queried one) is unused — `uploads_list.py` checks
   the returned value against the `video_id` being queried, so the wrapper
   returning `None` for a job that IS active would be wrong. Verify
   `uploads_list.py`'s call site before implementation.

4. **`current_stream_url.txt` consumers:** `ds-start.sh` reads this file. Confirm
   with the operator that `ds-start.sh` is not invoked in any other deploy path
   (e.g. a manual restart script) before deleting the write. If it is, add a
   one-line comment in the script noting the file is no longer updated.
